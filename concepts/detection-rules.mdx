## Overview

The detection engine runs every transaction through **103 production rules** across seven categories. Rules are evaluated in parallel; each triggering rule contributes its weight to the transaction risk score.

<Note>
  The rules engine is a platform-level component managed by Agregar. Institution users can see alert results but cannot view or modify rules directly.
</Note>

---

## Rule categories

| Category | Count | Focus |
|---|---|---|
| Amount | 6 | Threshold breaches, round numbers, CTR triggers |
| Velocity | 7 | Transaction frequency per customer, receiver, or device |
| Behavioral | 9 | Deviation from historical customer patterns |
| Network | 8 | Counterparty relationships, circular flows |
| Africa-specific | 8 | Mobile-money patterns specific to GH/NG/KE |
| Device fingerprint | 4 | Device sharing, SIM swap, IMEI change |
| Travel Rule | 1 | FATF R.16 threshold and originator/beneficiary fields |
| Shadow (pending promotion) | 8 | In A/B testing via shadow deployment |

---

## Rule lifecycle

```
DRAFT → SHADOW → PRODUCTION → ARCHIVED
```

| Status | Description |
|---|---|
| `DRAFT` | Created but not yet evaluated against transactions |
| `SHADOW` | Running in parallel; results logged but not counted toward score |
| `PRODUCTION` | Active; contributes to risk score |
| `ARCHIVED` | Disabled; kept for audit history |

Promotion from SHADOW → PRODUCTION requires a four-eyes approval (`APPROVE_ACTION` permission).

---

## Shadow deployment

New rules run in shadow mode alongside the production engine. The `ShadowComparisonLog` records results from both engines for every transaction, letting you validate a new rule for ≥14 days and ≥100 transactions before promoting.

## Amount rules (examples)

| Rule ID | Trigger |
|---|---|
| R-A01 | Amount exceeds the cash CTR threshold for the transaction's jurisdiction |
| R-A02 | Amount exceeds the STR reporting threshold |
| R-A03 | Round-number amount (ends in 000) |
| R-A04 | Amount exactly at a common smurfing boundary |
| R-A05 | Amount > 10× the customer's 30-day average |
| R-A06 | Zero-value transaction |

---

## Velocity rules (examples)

| Rule ID | Trigger |
|---|---|
| R-V01 | More than 10 transactions in 1 hour for the same customer |
| R-V02 | More than 5 transfers to the same receiver in 24 hours |
| R-V03 | More than 3 ATM withdrawals in 1 hour |
| R-V04 | Cross-border velocity: more than 3 transactions in 24 hours to different countries |
| R-V05 | Device used by more than 3 distinct customers in 24 hours |

---

## Africa-specific rules (examples)

| Rule ID | Trigger |
|---|---|
| R-AF01 | MoMo transfer to an account with no transactions in the past 30 days |
| R-AF02 | Agent cash-out above the agent's typical transaction limit |
| R-AF03 | Cross-border transfer to a non-ECOWAS country |
| R-AF04 | Wallet-to-wallet transfer immediately following a large cash deposit (layering signal) |

## Alert on rule trigger

When rules fire and the composite score crosses the FLAG threshold (≥ 90), an alert is created:

```json
{
  "alert_id": "ALT-001234",
  "transaction_id": "TXN-A1B2C3",
  "triggered_rules": ["R-V01", "R-A05"],
  "combined_risk_score": 112,
  "decision": "FLAG",
  "priority": "IMMEDIATE"
}
```
