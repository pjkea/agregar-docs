The Transaction Monitor pushes events to your registered HTTPS endpoint as they happen. Each institution registers its own URL — no shared global config, no polling.

---

## How it works

1. You register an endpoint URL via `POST /webhooks/endpoints`
2. We return a **signing secret** — store it securely, it's shown once
3. We POST a signed JSON envelope to your URL as events occur
4. You verify the signature, return 2xx within 10 seconds, process async
5. Failed deliveries retry 4 times with exponential back-off (30 s → 5 m → 30 m → 2 h)

---

## Register an endpoint

```bash
curl -X POST https://monitor.agregartech.com/api/v1/webhooks/endpoints \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://your-system.example.com/webhooks/tms",
    "events": ["TRANSACTION_SCORED", "ALERT_GENERATED", "STR_FILED"],
    "description": "Fidelity core banking integration"
  }'
```

**Response (201 Created):**

```json
{
  "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "url": "https://your-system.example.com/webhooks/tms",
  "events": ["TRANSACTION_SCORED", "ALERT_GENERATED", "STR_FILED"],
  "is_active": true,
  "description": "Fidelity core banking integration",
  "created_at": "2026-06-15T10:00:00Z",
  "secret": "a3f8c2d1e4b7..."
}
```

<Warning>
  The `secret` is returned **only at registration time**. Store it immediately in your secrets manager. To rotate it, delete the endpoint and re-register.
</Warning>

### Available event types

| Event | Fires when |
|---|---|
| `TRANSACTION_SCORED` | Every transaction finishes the full AML pipeline |
| `ALERT_GENERATED` | A high-priority (P1/Immediate) alert is created |
| `CASE_CREATED` | A case is opened from an alert |
| `CASE_ESCALATED` | A case is escalated |
| `SLA_BREACH` | A case exceeds its SLA threshold |
| `STR_FILED` | A Suspicious Transaction Report is filed |
| `CTR_FILED` | A Currency Transaction Report is filed |

Subscribe to only the events you need. You can update subscriptions at any time with `PATCH /webhooks/endpoints/{id}`.

## Event envelope

Every event uses the same outer shape:

```json
{
  "id": "evt_a1b2c3d4e5f6...",
  "event_type": "TRANSACTION_SCORED",
  "api_version": "2026-06-15",
  "created_at": "2026-06-15T10:30:00.123456Z",
  "livemode": true,
  "data": { ... }
}
```

Check `event_type` first; the structure of `data` depends on it.

## Verifying the signature

Every delivery includes an `X-TMS-Signature` header. Verify it before processing.

```
X-TMS-Signature: sha256=3c5e2a...
X-TMS-Event: TRANSACTION_SCORED
X-TMS-Delivery: evt_3fa85f64...
```

<CodeGroup>

```python Python
import hashlib, hmac
from fastapi import Request, HTTPException

async def verify_tms_webhook(request: Request, secret: str) -> dict:
    body = await request.body()
    sig_header = request.headers.get("X-TMS-Signature", "")

    if not sig_header.startswith("sha256="):
        raise HTTPException(status_code=401, detail="Missing signature")

    expected = hmac.new(secret.encode(), body, hashlib.sha256).hexdigest()
    received = sig_header[len("sha256="):]

    if not hmac.compare_digest(expected, received):
        raise HTTPException(status_code=401, detail="Invalid signature")

    return await request.json()
```

```javascript Node.js
const crypto = require('crypto');

function verifyTmsWebhook(body, signatureHeader, secret) {
  const expected = crypto
    .createHmac('sha256', secret)
    .update(body)           // raw Buffer, not parsed JSON
    .digest('hex');
  const received = signatureHeader.replace('sha256=', '');
  return crypto.timingSafeEqual(
    Buffer.from(expected),
    Buffer.from(received)
  );
}
```

</CodeGroup>

<Warning>
  Compute the HMAC over the **raw request body bytes**, not a re-serialised JSON string. Any whitespace difference will cause a mismatch.
</Warning>

## Delivery guarantees

| Property | Detail |
|---|---|
| Timeout | 10 seconds per attempt |
| Retry schedule | Immediate → 30 s → 5 min → 30 min → 2 h (max 5 attempts) |
| After 5 failures | Delivery marked `failed`; no further retries |
| Idempotency key | `id` field in the envelope (`evt_...`) — deduplicate on this |
| Ordering | No ordering guarantee across events; use `created_at` if sequence matters |

**Your endpoint should respond 2xx within 3 seconds** and process asynchronously. Any 2xx is accepted; we don't read the response body.

## Test your endpoint

Before going live, fire a test event to your URL:

```bash
curl -X POST https://monitor.agregartech.com/api/v1/webhooks/test \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "event_type": "TEST",
    "webhook_url": "https://your-system.example.com/webhooks/tms",
    "payload": { "hello": "world" }
  }'
```

The test payload is wrapped in the standard event envelope so your signature verification runs exactly as in production.

## Managing endpoints

```bash
# List all your registered endpoints
GET  /api/v1/webhooks/endpoints

# Pause deliveries without deleting
PATCH /api/v1/webhooks/endpoints/{id}
{ "is_active": false }

# Update subscribed events
PATCH /api/v1/webhooks/endpoints/{id}
{ "events": ["TRANSACTION_SCORED", "STR_FILED"] }

# Delete (all pending deliveries are cancelled)
DELETE /api/v1/webhooks/endpoints/{id}
```

## REST fallbacks

If your endpoint was unavailable and deliveries exhausted retries:

| Missed event | Fallback |
|---|---|
| `TRANSACTION_SCORED` | `GET /transactions/{transaction_id}` |
| `ALERT_GENERATED` | `GET /alerts?status=OPEN&priority=IMMEDIATE` |
| `STR_FILED` | `GET /filings?report_type=STR` |
| `CTR_FILED` | `GET /filings?report_type=CTR` |
| `CASE_ESCALATED` / `SLA_BREACH` | `GET /cases?status=ESCALATED` |
